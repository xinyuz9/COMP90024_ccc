# Part 1: “Big data” challenges and architectures

# 模块 1：大数据挑战与分布式数据库基础

## 1. 大数据的 "4V" 特征
![[Pasted image 20260504201954.png]]
提到“大数据”，人们往往只关注数据的“大”，但实际上它包含四个核心维度的挑战，即 4V 特征：

- **Volume（体量）**：数据的规模（如 Giga, Tera, Peta 级别）确实是一个重要标准，但这仅仅是其中一方面。
    
- **Velocity（速度）**：指新数据输入或更新到数据库的频率和极高速度。
    
- **Variety（多样性）**：指数据结构（Schema）的复杂性和多变性。在大数据场景下，数据通常来自不同的源头，拥有不同的结构，你无法像在传统关系型数据库中那样，依赖单一且整洁的数据表结构。
    
- **Veracity（真实性/准确度）**：指对数据准确度和来源的信任程度。由于数据来源广且大多是非结构化数据，数据源对同一事物的定义可能存在差异，且结构可能随时间推移而改变，导致数据可靠性降低。
    

> [!tip] 教授提示 教授在课上特别强调：“Big data is not about them being big. It's about more than that.”（大数据不仅仅关乎其规模）。尤其是 Velocity（更新插入速度）和 Variety（数据结构异构性）往往是驱动我们采用新架构的根本原因。

## 2. 为什么需要 NoSQL / 分布式数据库？

面对大数据的挑战，传统的解决方案开始显露局限性：

- **传统关系型数据库（Relational DBMS）的局限**：像 MySQL、PostgreSQL 这样的系统非常擅长保证数据一致性（Consistency），但它们依赖高度规范化（Normalized）的数据模型。在面对大数据的多样性（Variety）和真实性（Veracity）挑战时，严格的关系代数和规范化模型变得难以适用。
    ![[Pasted image 20260504202618.png]]
    
- **NoSQL (Not-Only SQL) 的崛起**：为了解决这些新问题，我们需要基于非关系型数据模型构建的数据库。这些系统在历史上通常使用其他查询语言，因此被称为 NoSQL。
    
    - 许多 NoSQL 系统依然支持 SQL，但功能受限（例如 ElasticSearch 的 SQL 就缺乏完整的 JOIN 支持）。
        
- **NewSQL 概念**：结合了传统关系型数据库的可靠性（Dependability）与 NoSQL 的可扩展性（Scalability）（例如 `CockroachDB`, `VoltDB`）。
    

> [!warning] 注意/易错点 **NoSQL 不等于“不能使用 SQL”**。它更准确的意思是 **"Not Only SQL"**。实际上在分布式环境中并没有限制使用 SQL 语言，只是我们经常需要借助其他更灵活的语言来处理复杂的数据结构。

## 3. 分布式环境下的三种主要数据模型

为了适应分布式计算节点，应对高并发和大数据流，目前主要采用以下三种数据模型：
![[Pasted image 20260504202724.png]]

- **键值存储 (Key-value store)**
    - **特点**：被剥离到最核心形态的数据库。通过一个 Key 来快速检索对应的 Value 块。
    - **优势与劣势**：速度极快，能够很好地应对高速度（Velocity）的数据流入；但查询语言非常简陋，不支持复杂的检索逻辑。
    - **代表技术**：Redis, RocksDB, Berkeley DB。
        
- **宽列存储 / 大表 (BigTable)**
    - **特点**：可以说是键值存储的进化版。将数据存储在按“列族 (Column families)”分组的列中。
    - **形象理解**：想象一个类似于电子表格的三维空间，大部分空间是空的（稀疏矩阵），同一行可以包含同一列族中的不同列。非常适合不遵循固定 Schema 的数据。
    - **代表技术**：Apache Cassandra, Apache Accumulo。
        
- **文档导向存储 (Document-oriented DBMS)**
    - **特点**：数据的基本存储单元是结构化文档（通常为 JSON 或 XML 格式），而不是数据表中的一行。
    - **优势**：文档内部可以包含嵌套的属性、数组等所有复杂信息。这意味着你查询某一实体时，所有相关信息都集中在一个文档中，完美契合分布式的存储需求。
    - **代表技术**：Apache CouchDB, MongoDB, **ElasticSearch**。
        

> [!info] 概念解析：为何文档数据库更适合分布式？ 在关系型数据库中，为了遵循“规范化（Normalization）”，一个人的信息可能被拆分在 Persons、Emails、PhoneNumbers 三张表里。而在分布式系统中，如果这三张表被分布在不同的物理机器上，要通过网络将它们重新拼接（Join）起来会非常低效。**文档数据库**则将同一个人的所有信息（如电话数组、邮箱数组）打包在一个单一的 JSON 文档中，这让分布式环境下的数据存取变得极其高效。

# 模块 2：数据库集群架构与 CAP 定理

## 1. 为什么需要数据库集群？
![[Pasted image 20260505113752.png]]
在分布式环境中，数据库集群（Clusters）是指一组分布在不同计算节点上的软件进程实例，但对用户而言，它们表现得就像一个单一的数据库。

- 我们需要集群主要是为了两个目的：一是将计算负载（Computing load）分配到多个节点以提升性能；二是存储同一数据的多个副本以实现冗余（Redundancy）。
    

> [!warning] 注意/易错点 仅仅在 Kubernetes 中把 Pod 的 `replicas` 设置为多个，并不意味着你的应用程序就神奇地变成了集群。对于带有状态（State）或Side effect 的软件（如数据库），因为每次操作都会改变数据库本身，实现集群化要复杂得多。

## 2. 三种经典的分布式架构对比

为了解决分布式存储问题，不同的数据库采取了截然不同的架构设计：

### A. CouchDB（完全去中心化）
![[Pasted image 20260505114313.png]]
- **节点平等**：集群中的所有节点都可以同时响应读取或写入请求（没有绝对的主节点）。
    
- **分片与冗余**：数据被划分为多个分片（Shards）并托管在所有节点上，每个分片都有多个副本。
    
- **高可用性**：如果一个节点宕机，由于其他节点上还保留有数据副本，剩余节点会继续工作。增加或移除节点时，分片会自动重新平衡。
    

### B. PostgreSQL (Federated 联邦架构)
![[Pasted image 20260505114933.png]]
- **单一入口**：这种架构由多个分离的数据库组成，但对外表现为一个入口。客户端请求通常只发送给主协调节点（Coordinator node）。
    
- **外部数据包装器（Foreign Data Wrapper）**：当请求的数据不在主节点而在其他节点时，主节点会通过此机制去请求对应节点。写入请求也会被转发，并且必须在所有相关节点上成功才算完成。
    
- **脆弱性**：如果其中一个节点宕机，由于部分数据变得不可访问，整个集群的运行可能会中止。
    

### C. ElasticSearch (主从架构)
![[Pasted image 20260505120915.png]]
- **角色分工**：ES 的节点具有不同的角色，最主要的是 **Master（主节点）** 和 **Data（数据节点）**。
    
- **主节点职权**：集群中只能有一个活跃的 Master 节点，负责协调集群级操作（如决定分片位置、创建/删除索引）。其他节点可以是“符合主节点资格（Master-eligible）”的备用节点。
    
- **读写机制**：写入（Update）请求只能发送给承载主分片（Primary Shard）**的节点，更改完成后再传播并等待所有**副本分片（Replica Shard）确认。而读取请求可以并行分发到副本分片以加快速度。
    

## 3. CAP 定理详解 (Brewer's CAP Theorem)


为什么会有上述截然不同的架构？这归因于分布式系统中一个无法完美解决的难题——**CAP 定理**。它指出，一个分布式数据库系统最多只能同时满足以下三项中的两项，三者的交集为空：

![[Pasted image 20260505123634.png]]
- **Consistency（一致性）**：所有收到响应的客户端，都会从集群的任意节点获得相同且非报错的响应数据。
    
- **Availability（可用性）**：每个客户端向集群发送请求，都会收到非报错的响应，但不保证每个节点返回的数据是最新或相同的。
    
- **Partition-tolerance（分区容错性）**：当集群中一个或多个节点无法与集群其余部分通信时（即发生网络分区），整个集群依然能够继续运行。
    

> [!tip] 教授提示 传统的 DBMS 架构不太关注网络分区，因为它们通常运行在单机或高质量的局域网中。但在云时代的数据中心里，由于节点分布在不同的机架或面临其他工作负载的干扰，超时导致的“软性分区（Soft partitions）”非常普遍。例如，ElasticSearch 只要主节点延迟 500 毫秒，就会判定为节点崩溃并触发重新选举。因此，在现代大数据架构中，**P（分区容错性）是必须考虑的前提，我们通常是在 C 和 A 之间做权衡**。

## 4. 应对网络分区的核心算法

面对 CAP 定理的权衡，不同的数据库采用了不同的算法机制：

### A. 两阶段提交 (Two-Phase Commit, 2PC) -> 追求 Consistency & Availability
![[Pasted image 20260505124018.png]]
- **机制**：主要用于关系型数据库（ElasticSearch 在一定程度上也使用）。它通过锁定数据（Locking data），将事务写入预写日志（Write-ahead logs），最后只有当集群所有节点都确认完成时，才正式提交事务。如果发生网络分区或任何节点未响应，则中止（Rollback）事务。
    
- **代价**：由于需要锁定数据，性能会下降；且在发生网络分区时，系统会完全失去可用性。
    

### B. Paxos / Raft -> 追求 Consistency & Partition-Tolerance
![[Pasted image 20260505125009.png]]
- **机制**：这是一类基于“共识（Consensus）”的算法。系统内包含提议者（Proposer）和接受者（Accepter）。提议只有在获得超过半数节点（Quorum）同意时才会被接受并写入。
	
- **1. 多数派写入 (Quorum / 法定人数)** 任何写入新数据的提案（Proposal），都必须获得集群中**过半数节点**的同意（Accept），才能正式落盘写入。
    
- **2. 旧时间戳拒绝机制 (Timestamp Rejection)** 如果某个节点“拒绝”了写入提案，通常不是因为“数据已经存在”，而是因为这个提案的**时间戳太旧**（该节点已经接收过发生得更晚、更新的提案）。
    
- **3. 网络分区时的“少数派罢工” (Partition Behavior)** 当网络断开、集群分裂时：
    
    - **多数派节点组**：由于还能凑够半数以上的票，会**继续正常接受写入请求**。
        
    - **少数派节点组**：由于永远凑不够法定人数，会**停止接受任何新的写入**（丧失写入可用性）。
        
    - _核心目的_：通过强迫被隔离的少数派“闭嘴”，彻底杜绝了集群分裂产生两套不同数据的风险（避免脑裂），从而完美保障了**一致性 (Consistency)**。

### C. 多版本并发控制 (Multi-Version Concurrency Control, MVCC) -> 追求 Availability & Partition-Tolerance
![[Pasted image 20260505125122.png]]
- **机制**：CouchDB 就是采用这种机制。它的工作原理类似代码管理工具 Git。每次并发更新都会生成不同的**版本号（Revision numbers）**，无需分布式锁定。
    
- **代价**：在网络分区期间，被隔离的节点仍会继续接受请求，保持极高的可用性。但在网络恢复后，可能会暴露出拥有相同版本号的冲突文档，此时一致性被打破，通常需要留给应用程序层的开发者来手动解决冲突。
    

![[Pasted image 20260505125159.png]]
> [!info] 总结：三种架构的 CAP 归属
> 
> 1. **CouchDB**：使用 MVCC，即使发生网络分区也能保持高度可用（AP），牺牲了一致性。
>     
> 2. **PostgreSQL** (Foreign Data Wrapper)：使用两阶段提交，不容忍网络分区（CA）。
>     
> 3. **ElasticSearch**：混合了用于复制数据的 2PC 和用于选举主节点的类 Paxos 算法。在发生网络分区时，可能因为无法选出主节点而停止接受写入（CP）。它通过引入**序列号（Sequence number）**和**主分片代号（Primary term）**来确保更新的顺序性。


## 5. 大数据存储的基础策略：分片与复制 (Sharding & Replication)

在应对大规模数据时，除了解决 CAP 的一致性问题，我们还需要解决物理存储容量和系统容错的问题。这就引入了两个核心操作：

- **分片 (Sharding)**：将数据“水平”切块。
    ![[Pasted image 20260505210336.png]]
    - **概念**：将数据库的行或文档划分为多个子集，分别存储在不同的服务器上。每个子集就是一个分片（Shard）。
        
    - **核心优势**：
        1. 通过将计算负载分布到多个节点来**提升性能**。
        2. 在集群扩容（添加新节点）时，更容易移动和重新平衡数据文件。
        3. 允许数据库的总数据量**突破单台机器的物理存储上限**。
            
    - **主要策略**：
        - **哈希分片 (Hash sharding)**：主要目标是将数据行**均匀地**分布到整个集群中。
        - **范围分片 (Range sharding)**：将相似或相关的数据（如来自同一地区的社媒帖子）集中存储在同一个分片上，方便针对性的批量查询。
            
- **复制 (Replication)**：为数据提供冗余。
    ![[Pasted image 20260505210423.png]]
    - **概念**：将同一行（或文档）拷贝并存储在不同的物理节点上，从而使数据库具备**容错能力 (Fault-tolerant)**。
        

> [!info] 概念解析：分片与复制的黄金组合 简单来说，**分片（Sharding）**是把庞大的数据切成不同的小块（Chunks）以解决容量瓶颈；而**复制（Replication）**是给这些小块做备份以防节点宕机。在实际应用中，这两者几乎总是**结合使用**的，以此来在保证最低数据安全级别的同时，最大化系统的可用性 (Availability)。

# 模块 3：ElasticSearch 核心概念与集群架构

## 1. 为什么在课程中选择 ElasticSearch？及其优缺点

ElasticSearch (ES) 最初是基于 Apache Lucene 构建的全文本搜索引擎，现已发展为支持分布式、文档驱动的大型 DBMS。

**ES 的主要优势 (Good at)：**
![[Pasted image 20260505132318.png]]
- **全文本搜索（Full-text search）**：继承自 Lucene，极其强大。
    
- **时间及空间序列数据**：非常擅长处理带有时间戳的计算机日志，以及支持按地理位置（空间坐标）查询。
    
- **非结构化数据处理**：原生支持 JSON 格式，容纳各种灵活结构的数据。
    
- **分布式架构优秀**：原生支持分片与复制，不仅可以横向扩展，而且能很好地运行在 Kubernetes 集群中。
    
- **支持向量数据存储**：这对机器学习和 K-近邻（KNN）查询非常有用。
    
- 具有丰富的生态系统（如 Kibana 图形化界面、HTTP API 交互）并且开源。
    

**ES 的劣势 (Bad at)：**
![[Pasted image 20260505133030.png]]
- **处理关联数据（Linked data）**：不支持传统意义上的多表 Join（除了特定的父子文档和 ES|QL 中的查找连接），对高度关联的数据还是关系型数据库更好。
    
- **多记录事务（Multiple-record transactions）**：ES 没有类似关系型数据库的锁机制，只能保证单条文档级别的原子性更新。如果你在更新一批文档时有其他进程同时修改了它们，就可能发生冲突。
    
- **多维度复杂分析**：不适合充当传统数据仓库（Data Warehouse）进行多维联机分析（OLAP）。
    

> [!warning] 注意/易错点：不要在 ES 中频繁修改数据 教授强调，ES 非常不擅长处理**频繁更改（Frequently changed）的数据。在 ES 中，文档一旦更新或删除，并不是在磁盘上原地修改，而是将其标记为已删除，并将新版本作为新文档追加（Appended）**到文件中。高频修改会导致磁盘上的索引体积飞速膨胀。

## 2. 核心术语梳理

理解 ES 必须掌握以下这组对应概念：
![[Pasted image 20260505133130.png]]
- **Index (索引)**：类比于关系型数据库中的“数据库”或“表”（介于两者之间）。它是同类文档的集合。
    
- **Document (文档)**：ES 中数据存储的最小原子单位，格式为 JSON 对象，类比于关系型数据库表中的“行（Row）”。
    
- **Data Stream (数据流)**：一组遵循相同命名模式的 Index 集合（例如将日志按天存储，命名为 `nginx-*`）。它允许你把跨越不同日期的多个索引作为一个整体来查询，极大方便了日志数据的管理。
    
- **Shard (分片)**：对 Index 的水平数据划分。通过哈希算法（Hash sharding）将文档均匀分布在不同分片上，也可以通过指定路由（Routing）将相关文档分到同一个分片。
    
- **Replica (副本)**：Shard 的备份数量。如果在建立时指定 Replica 为 2，则意味着同一个文档会有 3 份拷贝（1 个主分片 + 2 个副本分片）。
    

## 3. 节点 (Node) 角色与分工
![[Pasted image 20260505134453.png]]
在 ES 集群中，每个实例就是一个节点（Node），节点可被分配一种或多种角色：

- **Master (主节点)**：负责协调集群级别级别的操作，例如决定分片的位置、跟踪节点健康状态、创建或删除 Index。
    
    - 集群中可以有多个**具备主节点资格（Master-eligible）** 的节点，但**同一时间只有一个处于活跃（Active）状态的 Master**。
        
    - 当现任 Master 故障时，其余符合资格的节点将举行选举选出新的 Master。
        
- **Data (数据节点)**：负责实际管理和存储数据。如果一个数据节点没有主节点资格，它将只能管数据而不能参与 Master 选举。
    
- **Coordinating (协调)**：所有节点都能充当协调节点来接收客户端查询，并将计算分发下去。
    

## 4. 集群健康状态评估 (Green / Yellow / Red)

![[Pasted image 20260505135741.png]]
当你查看 ES 或 Kibana 时，经常会看到这三个颜色指标，它们用来衡量集群的运行状态：

- **Green (绿色)**：集群拥有足够的节点，完美承载了所有的主分片（Primary Shards）和副本分片（Replica Shards）。
    
- **Yellow (黄色)**：所有的主分片都正常工作（意味着没有数据丢失且数据库依然可用），但至少有一个**副本分片未被分配**。这通常是因为集群中的节点数少于你设定的总副本数（注意：ES 不允许将同一个分片的两个副本存放在同一个节点上）。
    
- **Red (红色)**：集群中的工作节点数量低于所需，导致部分**主分片无法正常分配**，部分数据不可用。
    

> [!tip] 教授提示：看到 Yellow 不要慌 教授在课上特意提到：“If your cluster is yellow, don't panic.”（看到黄色状态不要惊慌）。集群的状态是由其中“最差的”那个索引决定的。哪怕你的集群有 100 个索引，只要其中**有且仅有一个**索引是 Yellow（比如该索引配置了 3 个副本但只有 2 个节点），整个集群的状态就会显示为 Yellow。这不意味着集群崩溃，它仍然在正常工作，只是没有达到你预期的冗余安全级别。

## 5. 分片策略与索引生命周期 (Index Life-cycle)

### 分片数量设定的权衡
![[Pasted image 20260505135837.png]]
分片数量一旦在创建 Index 时确定便难以更改，因此需要谨慎评估：

- **大分片的缺点**：ES 在搜索时，每个分片占用一个线程（Thread）。单个分片越大，搜索速度越慢。
    
- **多分片的缺点**：虽然分片越多越能分散计算压力，但每个分片都是一个独立的进程，会消耗系统的 CPU、RAM 等资源。
    
- **经验法则 (Rule of thumb)**：建议每个分片容纳约 **2亿（200M）条文档**，或者将其大小控制在 **10GB - 50GB** 之间（取决于硬件）。
    
- **宁多勿少**：通常来说，分片设多一点比设少一点好，因为主分片的数量决定了你未来横向扩展添加节点的最大数量。
    

### 索引生命周期管理
![[Pasted image 20260505140007.png]]
对于大量时间敏感的数据（如日志），无限存储是不现实的。ES 提供生命周期策略（Lifecycle policies）来自动管理数据的衰退：

- 典型的生命周期阶段包含：**Hot -> Warm -> Cold -> Frozen -> Delete**。
    
- **应用实例**：例如设置 `nginx-*` 的策略为：前 10 天放在高性能的 Hot 硬件节点上处理高频写入；10天后迁移至低性能的 Warm 节点；30天后自动删除（Delete）以释放空间。

# 模块 4：ES 数据管理与类型

## 1. Index 映射 (Mappings)

在 ElasticSearch 中，映射（Mappings）就相当于关系型数据库中的表结构（Schema）。它负责定义 JSON 文档中的各个字段应该映射成什么数据类型，以及它们应该如何被索引。
![[Pasted image 20260505141700.png]]
映射主要分为两种方式：

- **动态映射 (Dynamic Mapping)**：如果你不提供明确的结构，ES 会根据传入文档的 JSON 内容自动推断并生成 Schema。
    
- **显式映射 (Explicit Mapping)**：由开发者预先定义好确切的字段和类型。
    

> [!warning] 注意/易错点：避免完全依赖动态映射 虽然动态映射很方便，但教授强烈建议使用**显式映射**。因为如果让 ES 自己做主，它会试图对所有字段进行“全文本索引”。把不需要搜索的字段建立全文本索引，将造成计算能力和磁盘空间的巨大浪费。你可以在映射中将 `dynamic` 参数设置为 `strict`，这样一旦传入的文档不符合预设结构，ES 就会直接拒绝该文档并报错。

### 更改已有的映射 (Reindexing)

一旦索引创建完成，你对映射的修改权限是非常有限的（通常只能增加新字段，不能修改或删除旧字段）。如果你必须彻底改变映射，唯一的方法是**重建索引（Reindex）**。 这通常意味着：使用新的映射创建一个新索引 -> 将老索引的数据迁移过去 -> 删除老索引。ES 为此提供了一个专门的 `_reindex` API 端点。

![[Pasted image 20260505141757.png]]

```JSON
POST _reindex
{
  "source": { "index": "my_index_v1" },
  "dest": { "index": "my_index_v2" }
}
```

## 2. 核心数据类型 (Data Types)

ES 支持非常丰富的数据类型，以满足非结构化和半结构化数据的存储需求：
![[Pasted image 20260505142248.png]]
- **Binary (二进制)**：被编码为 Base64 字符串。虽然能存图像等文件，但极为不推荐用 ES 存大体积文件。
    
- **Boolean (布尔)**：JSON 标准中的 `true` 或 `false`。
    
- **Numeric (数值)**：支持整型 (Integers)、浮点型 (Floats) 等。
    
- **Date (日期)**：可以是格式化的字符串时间戳，也可以是距离 1970 年的毫秒数（数值型）。
    
- **Object (对象)**：可以嵌套包含其他 JSON 属性的复杂对象。
    
- **Geo (地理空间)**：带有真实世界坐标的几何图形，常用于范围搜索。
    
- **Vector (向量)**：包含多维空间点坐标的数值数组，分为密集型（Dense）和稀疏型（Sparse），用于机器学习和相似度搜索。
    

### 重点对比：Text vs Keyword
![[Pasted image 20260505142337.png]]
对于字符串类型，ES 提供了两种截然不同的索引逻辑：

- **Text**：会被**全文本索引（Full-text indexed）**，适用于长文本，比如社交媒体帖子的具体内容。
    
- **Keyword**：适用于取值范围有限的分类数据，如邮政编码、城市名称等。它**不进行分词和全文本索引**，只能用于精确匹配查找和排序。
    

> [!tip] 教授提示：多重索引 (Multi-field indexing) 同一个字段其实可以同时被配置成多种类型。例如，一个“美剧单集名称”（如 _The Sopranos-S06E20-The Blue Comet_），你可以将其索引为 `keyword` 类型以便精确按剧集排序，同时再设定它映射为 `text` 类型，以便用户搜索里面的特定单词（如 "Comet"）。

### 关于数组 (Array)

严格来说，ES 并没有一种专门名为“数组”的数据类型定义。在 ES 中，**任何字段都可以包含零个、一个或多个相同类型的值**。比如你可以传一个包含多种语言名字的数组给 `name` 字段，前提是里面元素的类型一致。

## 3. 元数据字段 (Metadata Fields)
![[Pasted image 20260505143635.png]]
当你向 ES 插入一条文档时，除了原始数据，系统还会默认附带并生成一系列以下划线 `_` 开头的元数据字段：

- **`_index`**：表明该文档属于哪个索引。
    
- **`_id`**：文档的唯一标识符。插入时若不指定，ES 会自动生成一个。
    
- **`_source`**：你插入时的那个最原始的 JSON 文档内容。
    
- **`_version`**：文档的当前版本号。
    
- **`_score`**：这虽然在查询结果中返回，但并不是存储在硬盘里的。它表示该文档与当前查询语句的**匹配相关度**，分数越高越匹配。

# 模块 5：ES 数据查询与进阶特性（Queries & Advanced）

## 1. 四种查询语言概览 (Query Languages)

ElasticSearch (ES) 提供多种查询数据的方式，教授主要介绍了以下四种（外加 Kibana 自带的 KQL）：
![[Pasted image 20260505145506.png]]
- **Query DSL (领域特定语言)**
    - **特点**：ES 中最基础、使用最广、最强大的查询语言，是 Apache Lucene 查询语法的超集。
    - **语法**：完全以 JSON 格式编写。
    - **痛点**：对于人类来说极度“笨重（Unwieldy）”，写起来复杂且不直观。
        
- **ES SQL**
    - **特点**：对用户极其友好的 SQL 风格语法。
    - **局限**：它只实现了极其有限的 SQL 子集，比如**不支持多表直接 Join、不支持 DISTINCT 去重、不能返回包含多个值的字段数组、对地理位置查询的支持有限**。
        
- **EQL (Event Query Language)**
    - **特点**：专门用于查询具有时间相关性的“事件”，并寻找它们之间的关联。
    - **应用场景**：比如在网络安全或系统监控中，查询“在一个 1 分钟的移动时间窗口内，先后发生的几个报错日志”。
        
- **ES|QL (ElasticSearch Piped Query Language)**
    - **特点**：最新推出的管道查询语言。它类似于 SQL，但引入了管道符（`|`），前一步的输出作为下一步的输入。
    - **优势**：在查询过程中可以动态创建变量（如使用 `EVAL`）并进行数据转换，非常强大且易于追踪逻辑。
        

![[Pasted image 20260505151252.png]]
> [!tip] 教授提示：ES SQL 的局限与突破 因为 Query DSL 太难写，而 SQL 功能又不全，教授推荐了两个“走捷径”的办法：
> 
> 1. 利用 `/_sql/translate` 端点，把写好的 SQL 自动翻译成 Query DSL 格式，你再在它的基础上修改。
>     
> 2. 在你的 ES SQL 语句中，嵌套一个包含 Query DSL 的 `filter` 子句，从而实现 SQL 无法直接完成的操作（比如复杂的地理边界盒查询 `geo_shape`）。
>     


### Updating and Deleting
![[Pasted image 20260505152048.png]]

> [!info] 概念解析：并发控制的秘密武器 `_seq_no` 与 `_primary_term` 由于 ES 的分布式特性，不同节点的请求可能乱序到达。ES 不像关系型数据库那样有传统的锁机制，因此它利用 **`_primary_term`（标识当前主分片的届数）**和 **`_seq_no`（主分片内的变更序列号）**来实现乐观并发控制。
> 
> 在执行更新 (PUT) 或删除 (DELETE) 操作时，最好带上这两个值。如果此时其他客户端正好修改了这条数据（序列号变了），ES 会返回 **409 Conflict（冲突）** 错误，以防止你误覆盖别人的最新更改。

## 2. 核心搜索场景与案例解析 (Search Scenarios)

ElasticSearch 最强大的地方在于它能够处理多种维度的搜索请求。以下是教授在课件中重点演示的几种核心搜索模式：

### Multi-Index at once
![[Pasted image 20260505152359.png]]
### A. 多条件组合搜索 (Multiple Criteria Search)

在 Query DSL 中，我们通常使用 `bool`（布尔查询）来组合多个条件。其中最核心的两个逻辑键是 `must` 和 `should`。

- **`must` (相当于逻辑 AND)**：数组中的所有条件**必须同时满足**。
    
    - _Slide 案例_：查询分数 $\ge 80$ 且名字以 "john" 开头的学生。
        
    ```JSON
    POST /students/_search
    {
      "query": {
        "bool": {
          "must": [
            { "range": { "mark": { "gte": 80 } } },
            { "match": { "name": "john *" } }
          ]
        }
      }
    }
    ```
    
- **`should` (相当于逻辑 OR)**：数组中的条件满足**任意一个**即可。
    ![[Pasted image 20260505151605.png]]
    - _Slide 案例_：查询分数 $\ge 90$ 或者名字以 "john" 开头的学生。只需将上述外层的 `must` 替换为 `should`。
        

> [!info] 概念解析：大小写敏感性 在默认情况下，Query DSL 的 `match` 对文本（Text）字段的搜索是**不区分大小写**的（Case-insensitive）。

### B. 文本搜索 vs 关键字搜索 (Text vs Keyword)

- **Text 搜索**：用于全文本匹配。例如在 SQL 中使用 `MATCH` 函数寻找包含 "Windows" 的日志消息。
![[Pasted image 20260505152955.png]]
```sql
SELECT data.status, data.type 
FROM "filebeat-8.7.1" 
WHERE MATCH('message', 'Windows')
```

*   **Keyword 搜索**：用于精确匹配分类数据。
    *   *教授提示*：Keyword 是区分大小写的（"Luca" 和 "luca" 是两个不同的词）。为了避免遗漏，通常配合 `LCASE()`（转小写）函数来查询。

![[Pasted image 20260505153017.png]]
```sql
SELECT data.status 
FROM "filebeat-8.7.1" 
WHERE LCASE(data.type) = 'chatcompletions-request'
```

### C. 空间/地理位置搜索 (Geographic Search)
寻找处于特定坐标范围（Bounding box）内的文档。
![[Pasted image 20260505152824.png]]
*   由于 ES SQL 原生对地理查询支持很弱，教授演示了一个非常实用的“嫁接”技巧：**在 SQL 查询的请求体中，利用 `filter` 字段直接插入 Query DSL 代码**。
    
```json
    POST /_sql?format=text
    {
      "query": "SELECT data.status, data.type FROM \"filebeat-8.7.1\"",
      "filter": {
        "geo_shape": {
          "location": {
            "shape": {
              "type": "envelope",
              "coordinates": [ [13.0, -53.0], [14.0, -52.0] ]
            },
            "relation": "within"
          }
        }
      }
    }
```

### D. 时间搜索 (Time Search)
ES SQL 提供了极其灵活的时间函数，常用于日志分析。
![[Pasted image 20260505152849.png]]
*   *Slide 案例*：查询过去两个月内的观察记录，可以使用 `TODAY() - INTERVAL '2' MONTHS`

## 2. 处理海量数据：分页 (Pagination)
默认情况下，ES 每次查询只返回 1000 条结果。如果要获取后续数据，必须使用**游标 (Cursor)**

*   **工作流**：
	* 第一次查询后，ES 会在 JSON 底部返回一个很长的 `cursor` 字符串（类似书签）
		![[Pasted image 20260505153223.png]]
	* 第二次查询时，你不需要再发送原始的 `query`，只需发送这个 `cursor`
    
```json
    POST /_sql?format=json
    {
      "cursor": "t8jsAORGTACkUluO0zAUtdu..."
    }
```
*   **终止条件**：当返回的 `cursor` 为空时，说明所有页都已读完。

> [!warning] 易错点：游标超时
> 游标是有时间敏感性的（Time-sensitive），它并不是永久存储的。如果你隔了几分钟才去请求下一页，游标会失效，你只能从第一页重新查起。

## 3. 数据统计：聚合与透视 (Aggregations & PIVOT)

- **指标聚合 (Metric Aggregation)**：使用常规的 SQL 聚合函数（如 `COUNT`, `SUM`）对数据进行基础统计，通常配合 `GROUP BY` 使用。
    ![[Pasted image 20260505155013.png]]
	- _Slide 案例 1（计数）_：按年份和状态分组，统计过去 4 年的日志数量。
    - _Slide 案例 2（求和）_：按状态分组，计算 `payload_size` 的总和。
	
*   **桶聚合 (Bucket Aggregation)**：根据特定条件（如时间间隔）将数据分组，就像做直方图
	![[Pasted image 20260505155123.png]]
    *   *Slide 案例*：以 1 年为间隔（`INTERVAL 1 YEAR`）对数据进行分组统计数量
    
*   **数据透视 (PIVOT - 模拟 OLAP)**：这是 ES SQL 的高级特性，可以将**行数据转换为列数据**
    *   *Slide 案例*：根据相对湿度（65.0, 66.0, 67.0）作为不同的列，分别计算每个监测站的平均气温
    ![[Pasted image 20260505160003.png]]
```sql
    SELECT * FROM (SELECT rel_hum, stationid, air_temp FROM observations) 
    PIVOT (AVG(air_temp) FOR rel_hum IN (65.0, 66.0, 67.0))
```

## 4. ES|QL 语法解析 (管道查询语言)
![[Pasted image 20260505161448.png]]
ES|QL 是 ES 最新的语言，采用类似 Unix 的**管道符（`|`）**，将上一步的输出作为下一步的输入。这使得复杂的数据转换逻辑变得极其清晰。
![[Pasted image 20260505173750.png]]
*   *Slide 实战代码剖析*：
    
```sql
    FROM logs
    | WHERE level == "ERROR" AND duration_ms > 500  -- 第一步：过滤出大于500ms的错误日志
    | EVAL duration_sec = duration_ms / 1000.0      -- 第二步：新建一个变量，将毫秒转换为秒
    | STATS count(*) AS error_count BY service      -- 第三步：按 service 字段进行分组聚合
    | SORT error_count DESC                         -- 第四步：按错误数量降序排列
    | LIMIT 5                                       -- 第五步：只取前5名
```

> [!warning] 注意
> 在 ES|QL 中，双等号 `==` 是**比较运算符**（判断是否相等），而单等号 `=` 是**赋值运算符**（用于 `EVAL` 创建变量）。这与传统 SQL 不同。

## 5. 突破限制：在 ES 中实现数据连接 (Joins)
ES 不擅长多对多的关系映射，但提供了三种受限的 Join 方法：
![[Pasted image 20260505173817.png]]
### A. 父子关系 (Parent-child) -> 同一索引内部连接
![[Pasted image 20260505173849.png]]
*   必须在 Mapping 中预先声明 `relation_type`。
	![[Pasted image 20260505174120.png]]
*   插入子文档时，必须指明其 Parent ID，且两者**必须在同一个分片（Shard）上**
	![[Pasted image 20260505174222.png]]
*   查询极其笨重，需使用 Query DSL 的 `has_parent` 语法

### B. 查找连接 (Lookup Joins) -> ES|QL 跨索引连接
*   *Slide 案例*：给日志表补充用户信息。
    ![[Pasted image 20260505174206.png]]
```sql
    FROM logs
    | LOOKUP JOIN users ON user_id
    | KEEP @timestamp, user_id, user_name, message
```
*   **严苛限制**：被 Lookup 的那个表（如上例的 `users`）必须是**只有一个分片（Unsharded）**的，并且总体积必须足够小，能被完全装进主节点的 Heap Memory（堆内存）中

## 6. 前沿特性：向量数据查询 (Vector Data & KNN)
![[Pasted image 20260505174339.png]]
通过 KNN（K-近邻）搜索和 HNSW 算法，ES 可以在多维空间中寻找最相似的数据点，常用于大模型（LLM）的**检索增强生成 (RAG)**。
![[Pasted image 20260505174408.png]]
*   *Slide 案例解析*：
    ![[Pasted image 20260505174450.png]]
```json
    POST byte-image-index/_search
    {
      "knn": {
        "field": "byte-image-vector",
        "query_vector": [-5, 9],    // 这是你提问转化出的向量坐标
        "k": 10,                    // 最终希望返回的最相似的10个文档
        "num_candidates": 100       // 每个分片只取前100个候选者进行全局对比，以提升性能
      }
    }
```
